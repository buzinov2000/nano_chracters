# Character Pipeline Bot

Телеграм-бот для автоматизации нейтральной части пайплайна разработки персонажей.

---

## Контекст и задача

Рабочий флоу разработки персонажей состоит из двух частей:

**Творческая (остаётся у человека):**
- Отбор скетчей
- Формулировка гипотез и направлений
- Оценка результатов
- Принятие решений

**Нейтральная (автоматизируется ботом):**
- Перевод гипотезы в промпт для генератора
- Отправка промпта в Gemini Image API
- Возврат сетки вариантов с номерами
- Микро-подсказки для следующей итерации
- Итерация промпта по обратной связи

---

## Архитектура

```
[пользователь: скетч + рефы (опционально) + текст]
        ↓
[Telegram Bot — курьер, своей обработки нет]
        ↓
[Gemini Flash — промпт-агент]
  получает: скетч + рефы + гипотеза пользователя
  отдаёт: промпт для генератора + 2-3 подсказки
        ↓
[Gemini Image API — генерация 4 вариантов параллельно]
        ↓
[бот возвращает превью сеткой с номерами]
[бот возвращает подсказки для следующей итерации]
        ↓
[пользователь: "1 3" → полное разрешение]
[пользователь: /промпт правки → новая итерация]
```

**Арт-директор — отдельно, не в боте:**
Живёт в Claude Projects с брифами проекта в файлах. Вызывается вручную когда нужна
экспертная оценка. Уже оплачен в рамках подписки Claude.

## Агенты

### Agent 1 — Промпт-агент (Gemini Flash)
- **Вход:** картинка скетча + референсные изображения (опционально) + описание гипотезы человеческим языком
- **Задача:** перевести в развёрнутый промпт для генератора
- **Дополнительно:** после промпта предлагает 2–3 микро-подсказки для следующей итерации
  (например: "Попробовать сетку 2×2 с разными материалами?", "Сделать версию в стиле мягкой игрушки?")
- **Контекст:** кастомная инструкция под стиль студии (3D, материалы, объём)
- **Модель:** Gemini 2.5 Flash (дешевле Haiku, нативно мультимодален, один провайдер)

### Арт-директор — вне бота
- **Где:** отдельный Project в Claude (уже оплачен)
- **Как:** бриф и референсы хранятся в файлах проекта
- **Вызов:** вручную, когда нужна экспертная оценка вариантов
- **Модель:** Claude Sonnet 4.6 (через существующую подписку)

## Интерфейс бота — команды

| Команда | Действие |
|--------|----------|
| [скетч + текст] | запуск нового цикла генерации |
| [несколько фото + текст] | первое фото = скетч, остальные = рефы |
| `/start` | начать новый проект, сбросить контекст |
| `/prompt` | показать текущий промпт текстом |
| `/prompt_edit` | редактировать промпт (следующее сообщение = правки) |
| `/model` | переключить модель генерации (inline-кнопки) |
| `/more` | сгенерировать ещё 2 варианта |
| Inline-кнопки 1-4 | получить вариант в полном разрешении (как документ) |
| Inline-кнопка «Ещё 2» | генерация доп. вариантов |

> **Кириллические алиасы:** `/старт` и `/промпт` обрабатываются через текстовый хэндлер, т.к. Telegram Bot API не поддерживает не-латинские команды в `CommandHandler`.
>
> **Меню бота:** команды зарегистрированы через `set_my_commands` и видны в меню Telegram.

## Модели генерации

| Модель | Model ID | Режим | Разрешение | Таймаут |
|--------|----------|-------|------------|---------|
| Nano Banana (быстрая) | `gemini-2.5-flash-image` | 4 параллельных вызова | 1K | 300 сек |
| Nano Banana Pro | `gemini-3-pro-image-preview` | Grid 2x2 (1 вызов) | 2K | 300 сек |
| Nano Banana 2 (качество) | `gemini-3.1-flash-image-preview` | Grid 2x2 (1 вызов) | 2K | 600 сек |

- Переключение через `/model`, выбор сохраняется между перезапусками бота (`user_data.json`)
- **Grid 2x2:** для Pro и Quality моделей промпт-агент использует отдельный системный промпт (`prompt_agent_grid.txt`), который включает инструкцию генерировать сетку 2x2. Модель возвращает одно изображение, которое разрезается на 4 варианта через Pillow
- **Aspect ratio:** не задаётся принудительно — модель сама определяет ориентацию по входному скетчу
- `thinking_level` НЕ поддерживается image-моделями (только текстовыми)

## Заметки

- Бот хранит картинки в памяти процесса (в сессии) — при перезапуске сбрасывается. Дефолтная модель сохраняется в `user_data.json`
- Нагрузка на VPS минимальная, вся обработка на стороне Google AI API
- Gemini Image возвращает одну картинку за вызов — для fast-модели делается N параллельных async-вызовов, для Pro/Quality — один вызов с grid 2x2
- Один провайдер (Google AI) = один ключ = меньше сложности
- `thinking_level` не поддерживается image-моделями — только текстовыми (gemini-2.5-flash и т.д.)
- `aspect_ratio` в ImageConfig: поддерживаемые значения `"1:1"`, `"2:3"`, `"3:2"`, `"3:4"`, `"4:3"`, `"9:16"`, `"16:9"`. При `None` (default) API ставит 1:1. Мы не задаём — модель ориентируется по скетчу
- Telegram httpx таймауты увеличены: read=60, write=60, connect=30 (дефолтные слишком маленькие для отправки больших файлов)
- **Сессии привязаны к `user_id`**, а не к `chat_id` — в групповом чате каждый пользователь имеет свою сессию, lock и лимит
- **`concurrent_updates=True`** — необходимо для работы session lock и параллельной обработки нескольких пользователей
- **Семафор** (`MAX_CONCURRENT_API_CALLS=8`) ограничивает параллельные вызовы к Image API, предотвращая 429 от Google
- **Дневной лимит** живёт в памяти — при перезапуске бота сбрасывается
- **Группы:** для работы в групповых чатах нужно отключить Privacy Mode через @BotFather (`/setprivacy` → Turn off), иначе бот не видит обычные сообщения

## Известные особенности и решённые проблемы

- **Кириллические команды:** Telegram `CommandHandler` не поддерживает не-ASCII символы. `/старт` и `/промпт` обрабатываются через `handle_text`
- **Grid 2x3 вместо 2x2:** без строгой инструкции в промпте модель может генерить 6 картинок. Решено через отдельный системный промпт для grid-моделей
- **Inline-кнопки блокируют друг друга:** последовательная обработка callback_query тормозила отправку. Решено через `asyncio.create_task()` для фоновой отправки документов
- **Model ID:** `gemini-3.0-pro-image` → правильный ID `gemini-3-pro-image-preview`
- **Медиагруппы:** Telegram отправляет несколько фото как отдельные update. Буферизация через `job_queue.run_once` с задержкой 1.5 сек
